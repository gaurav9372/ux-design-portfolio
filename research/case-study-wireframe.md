# UI/UX Case Study Page — Wireframe Architecture

A section-by-section blueprint for building case study pages that hiring managers actually read. Based on research from Behance, Dribbble, UXfol.io, UXPin, IxDF, and hiring manager interviews.

---

## Page Structure (15 sections)

```
1.  Hero
2.  Project Overview
3.  The Problem
4.  Goals & Success Metrics
5.  Research & Discovery
6.  User Personas
7.  User Journeys & IA
8.  Wireframing & Ideation
9.  Usability Testing
10. Visual Design System
11. Final Screens
12. Results & Impact
13. Reflections
14. Client Testimonial
15. Next Project Nav
```

## 1. Hero

**Purpose:** Hook the reader in 3 seconds. Lead with the outcome, not the process.

```
┌─────────────────────────────────────────────────┐
│  [Client Logo]                                  │
│                                                 │
│  PROJECT TITLE                                  │
│  "Redesigning the onboarding flow that          │
│   boosted conversions by 35%"                   │
│                                                 │
│  ┌─────────────────────────────────────────┐    │
│  │                                         │    │
│  │         Hero Mockup / Key Screen        │    │
│  │         (highest fidelity visual)       │    │
│  │                                         │    │
│  └─────────────────────────────────────────┘    │
│                                                 │
│  Role        Timeline     Team       Platform   │
│  UX Design   8 weeks      4 people   iOS/Web    │
└─────────────────────────────────────────────────┘
```

**Content:**
- Project title (outcome-focused, not just the product name)
- One-line subtitle with the key result
- Hero image — the most impressive screen/mockup
- Metadata strip: Role, Duration, Team Size, Tools, Platform

**Why it matters:** Hiring managers decide in seconds whether to keep scrolling. Lead with impact.

## 3. The Problem

**Purpose:** Create tension. Show you understood the user and business pain before designing.

```
┌─────────────────────────────────────────────────┐
│  THE PROBLEM                                    │
│                                                 │
│  [Business context — why this project existed]  │
│                                                 │
│  [User pain points — what was broken]           │
│                                                 │
│  [Constraints — budget, timeline, tech limits]  │
│                                                 │
│  Problem Statement:                             │
│  "Users were abandoning the checkout flow at    │
│   step 3 because..."                            │
│                                                 │
│  ┌─────────────────────────────────────────┐    │
│  │  "Before" screenshot or data showing    │    │
│  │   the broken state                      │    │
│  └─────────────────────────────────────────┘    │
└─────────────────────────────────────────────────┘
```

**Content:**
- Business context (why this project existed)
- User pain points (what was broken for real people)
- Constraints (budget, timeline, technical limitations)
- One clear problem statement sentence
- Optional: "Before" screenshots or analytics showing the broken state

## 4. Goals & Success Metrics

**Purpose:** Show strategic thinking — you defined what success looks like before designing.

```
┌─────────────────────────────────────────────────┐
│  GOALS                                          │
│                                                 │
│  Business Goals        User Goals               │
│  ● Increase revenue    ● Find products faster   │
│  ● Reduce support      ● Complete checkout      │
│    tickets               in < 2 minutes         │
│                                                 │
│  Design Goals          Success Metrics          │
│  ● Simplify flow       ● Conversion rate +20%   │
│  ● Reduce cognitive    ● Task completion > 90%  │
│    load                ● Support tickets -30%   │
└─────────────────────────────────────────────────┘
```

**Content:**
- Business goals (what the company needed)
- User goals (what users needed)
- Design goals (what the design needed to achieve)
- Success metrics defined upfront (what you planned to measure)

## 5. Research & Discovery

**Purpose:** Prove you're research-driven, not assumption-driven. This is where senior designers separate themselves.

```
┌─────────────────────────────────────────────────┐
│  RESEARCH & DISCOVERY                           │
│                                                 │
│  Methods Used:                                  │
│  ┌──────────┐ ┌──────────┐ ┌──────────────┐    │
│  │ User     │ │ Competi- │ │ Analytics    │    │
│  │Interviews│ │tive Audit│ │ Review       │    │
│  │ (8 users)│ │(5 brands)│ │ (3 months)   │    │
│  └──────────┘ └──────────┘ └──────────────┘    │
│                                                 │
│  Key Findings:                                  │
│  1. 67% of users couldn't find...              │
│  2. Competitors solved X by...                  │
│  3. Analytics showed drop-off at...             │
│                                                 │
│  "I didn't expect the checkout to be            │
│   this confusing" — User 4, Interview           │
│                                                 │
│  ┌─────────────────────────────────────────┐    │
│  │  [Affinity diagram / Competitive        │    │
│  │   audit matrix / Empathy map]           │    │
│  └─────────────────────────────────────────┘    │
└─────────────────────────────────────────────────┘
```

**Content:**
- Research methods used and WHY you chose them
- Key findings as numbered takeaways
- 1-2 direct user quotes
- Research artifacts: empathy maps, affinity diagrams, competitive audit
- Data visualizations of findings

## 7. User Journeys & Information Architecture

**Purpose:** Show you mapped the experience before jumping into screens.

```
┌─────────────────────────────────────────────────┐
│  USER JOURNEY                                   │
│                                                 │
│  Current State (pain points marked in red):     │
│  ┌─────────────────────────────────────────┐    │
│  │ Discover → Browse → [!]Cart → [!]Pay → │    │
│  │ [!]Confirm → Receive                   │    │
│  └─────────────────────────────────────────┘    │
│                                                 │
│  Redesigned Flow:                               │
│  ┌─────────────────────────────────────────┐    │
│  │ Discover → Browse → Quick Pay → Done    │    │
│  └─────────────────────────────────────────┘    │
│                                                 │
│  INFORMATION ARCHITECTURE                       │
│  ┌─────────────────────────────────────────┐    │
│  │         [Sitemap / IA diagram]          │    │
│  └─────────────────────────────────────────┘    │
│                                                 │
│  KEY USER FLOWS                                 │
│  ┌─────────────────────────────────────────┐    │
│  │   [Task flow for primary user goal]     │    │
│  └─────────────────────────────────────────┘    │
└─────────────────────────────────────────────────┘
```

**Content:**
- Current-state journey map (pain points highlighted)
- Future-state journey map (improvements highlighted)
- Sitemap / information architecture diagram
- User flows for 1-2 key tasks

## 8. Wireframing & Ideation

**Purpose:** Show your thinking process. Messy sketches > polished output here.

```
┌─────────────────────────────────────────────────┐
│  IDEATION & WIREFRAMING                         │
│                                                 │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐        │
│  │ Sketch 1 │ │ Sketch 2 │ │ Sketch 3 │        │
│  │ (photo)  │ │ (photo)  │ │ (photo)  │        │
│  └──────────┘ └──────────┘ └──────────┘        │
│  Early explorations on paper                    │
│                                                 │
│  Low-fidelity wireframes:                       │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐        │
│  │ Screen A │ │ Screen B │ │ Screen C │        │
│  │ v1       │ │ v1       │ │ v1       │        │
│  └──────────┘ └──────────┘ └──────────┘        │
│                                                 │
│  Why this layout?                               │
│  [Annotation explaining the key decision]       │
│                                                 │
│  ✗ APPROACH THAT FAILED:                        │
│  [Screenshot + why it didn't work]              │
│  → What we learned from this failure            │
└─────────────────────────────────────────────────┘
```

**Content:**
- Photos of sketches, whiteboard sessions
- Low-fidelity wireframes with annotations
- Iterations showing evolution (v1 → v2 → v3)
- Key design decisions explained (WHY, not just what)
- Failed approaches and why they were abandoned (this is gold for hiring managers)

## 9. Usability Testing

**Purpose:** Prove you validate with real users, not just stakeholder opinions.

```
┌─────────────────────────────────────────────────┐
│  USABILITY TESTING                              │
│                                                 │
│  Method: Remote unmoderated (Maze)              │
│  Participants: 6 users matching personas        │
│  Tasks: 3 key scenarios                         │
│                                                 │
│  Results:                                       │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐        │
│  │ Task 1   │ │ Task 2   │ │ Task 3   │        │
│  │ 92% ✓    │ │ 58% ✗    │ │ 85% ✓    │        │
│  │ success  │ │ success  │ │ success  │        │
│  └──────────┘ └──────────┘ └──────────┘        │
│                                                 │
│  BEFORE → AFTER (based on testing feedback):    │
│  ┌──────────┐    ┌──────────┐                   │
│  │ Old      │ →  │ New      │                   │
│  │ Screen   │    │ Screen   │                   │
│  │ (failed) │    │ (passed) │                   │
│  └──────────┘    └──────────┘                   │
│  [What changed and why]                         │
└─────────────────────────────────────────────────┘
```

**Content:**
- Testing methodology (remote/in-person, sample size, tools)
- Task scenarios tested
- Success/failure rates per task
- Before/after comparisons showing iterations from feedback
- What broke and how you fixed it

## Quick Reference — What Hiring Managers Look For

| Priority | What they check | Where it lives |
|----------|----------------|----------------|
| 1 | Measurable results | Section 12 |
| 2 | Your specific role & contribution | Section 2 |
| 3 | Research backing decisions | Section 5 |
| 4 | Iteration from testing | Section 9 |
| 5 | Visual craft | Sections 10-11 |
| 6 | Self-awareness | Section 13 |
| 7 | Problem framing | Section 3 |

## Recommended Length

- **Total:** 800-1,500 words + visuals
- **Sections:** 10-15 max
- **Images:** 60-70% of page area
- **Read time:** 5-8 minutes
- **Format:** Scrollable single page, not slides
