Design System — Audit & Proposal

From scattered rules
to one visual language.

This document audits the current state of the portfolio's design tokens and proposes a cleaner, more consistent system. Left side = what we have today. Right side = what we should move to. Once approved, these tokens will be rolled across every page.

All previews are live CSS — what you see is what we'll ship. Works on desktop, tablet, and mobile.

01 — Colors

Color tokens

A portfolio rarely needs more than 10 color tokens. More than that creates ambiguity — "which gray do I use here?"

Current — 20+ values

Too many colors, unclear names

Variables split by numbered suffix (bg, bg2, bg3, bg4). Two "muted" tokens with overlapping usage. Hardcoded hex values scattered in page CSS.

Backgrounds — 4 tokens

--bg#121715
--bg2#1c2320
--bg3#181e1b
--bg4#0e1210

Text — 2 muted tokens

--ink#ffffff
--mutedwhite/64%
--muted2white/92%

Semantic (scattered, not variables)

Error A#ff6b6b
Error B#e74c3c
Success#2ecc71
Issues: Four near-identical dark backgrounds (all look the same on-screen). Two muted text tokens used interchangeably. Error red appears as two different hex values depending on the file. No semantic color variables for states.
Improved — 11 tokens

Semantic names, clear hierarchy

Each token has a single purpose. Names describe intent ("text-secondary"), not position ("muted2"). Backgrounds collapsed to 3 levels that actually differ visually.

Surface — 3 levels

--surface-base#0F1412Page background
--surface-raised#1A201DSections, cards
--surface-overlay#252B28Modals, nav blur

Text — 3 tiers of emphasis

--text-primary#FFFFFFHeadings, body
--text-secondarywhite/72%Descriptions
--text-tertiarywhite/48%Metadata, labels

Brand — accent only

--accent#FFC321Brand, CTAs
--accent-hover#FFD658Hover states

Borders — 2 levels

--border-subtlewhite/8%Divider lines
--border-defaultwhite/14%Inputs, cards

Semantic states

--state-error#FF6B6B
--state-success#34D399
Rationale: 11 tokens total vs 20+. Each is used for one purpose. The three surface levels have noticeable contrast between them (so raised cards actually look raised). Error and success are tokenized so they never get hardcoded again.

02 — Typography

Type scale

A predictable modular scale reduces decisions and makes every page feel related. The current site has 17+ distinct font sizes across pages — we can cut that in half without losing expression.

Current — 3 fonts, 17+ sizes

Three fonts, overlapping scales

Heebo and Outfit serve nearly the same role. Arbitrary sizes like 44px, 36px, 32px, 28px appear on different pages for the "same" heading level.

Hero I Design Outfit 500 / 72px
Hero alt Care Naturals Heebo 800 / 52px

H2 (B) Section Title Heebo 800 / 36px
H3 Card Title Outfit 500 / 32px
Body L Paragraph text example Inter 500 / 18px
Body Default body text Inter 500 / 16px
Caption Eyebrow Label Inter 500 / 11px
Issues: Heebo and Outfit do similar jobs — using both increases page weight and visual noise. H2 appears at 44px, 36px, 32px, 28px across pages with no rule. Body weight defaults to 500 (medium), which makes long-form reading feel heavier than it should.
Improved — 2 fonts, 9 sizes

Two fonts, modular scale (1.25 ratio)

Remove Heebo. Outfit handles all headings. Inter handles all body. 9 sizes cover every use case, each built on the perfect-fourth modular scale (1.25x).

Display Display Outfit 500 / 72px
H1 Page Title Outfit 500 / 56px
H2 Section Outfit 500 / 44px

Body Default paragraph text Inter 400 / 16px
Small Metadata, captions Inter 500 / 14px
Eyebrow Eyebrow Label Outfit 500 / 12px
Rationale: Body defaults to weight 400 (regular) for better readability on long text. Medium (500) is reserved for UI, headings, and small/caption text where extra weight aids legibility. Removing Heebo saves ~35KB of font weight and simplifies visual identity.

03 — Border Radius

Roundness scale

The current site uses 8+ different radius values including leftover `--radius: 10px` that's almost never applied. A dark editorial site should commit to one sharp-or-soft direction.

Current — 8 values

No consistent language

Cards use 0, buttons use 12px, inputs use 12px, some cards use 16px, pills use 999px. Feels unintentional.

Improved — 2 values only

Sharp corners, period.

No rounded rectangles anywhere — buttons, cards, inputs, chips, modals all use 0. The only radius allowed is --radius-full (999px) for inherently circular shapes (avatars, color swatches, dot indicators).

--radius-none0Everything — buttons, cards, inputs, images, modals, filter chips, badges, section panels

Rationale: Editorial identity is built on confident geometry. Rounded rectangles dilute that tone. By committing to 0 for every rectangle, the site feels like it was designed on paper with a straight-edge — every corner is intentional. Circles are reserved for things that are genuinely circular in the real world.

05 — Buttons

Button system

The site currently has at least 8 distinct button styles — some are nearly identical but implemented separately. A clean system has one primary, one secondary, and one tertiary (text/link) variant.

Current — 8+ variants

Too many one-off styles

Each page invented its own CTA. Inconsistent padding, border-radius, uppercase rules, and hover behaviors.

Pill primary (projects page)

Pill 999px · 14px 28px · filled · dark text

Pill 100px · 10px 20px · transparent · outlined

Nav CTA

Custom uppercase · no border

Square (burger/toggle)

Badge

Pill · 4px 12px · 11px

Design
Issues: Each button style is bespoke. A user can't predict which interaction affords what. The pill-primary and pill-outline both live on pages but look so different they feel like different products.
Improved — text-only, one system

No boxes. Letters + motion.

One CTA pattern across the site: uppercase text with a trailing arrow. Hover triggers the per-letter flip (white → accent yellow) with staggered delay. No backgrounds, no borders, no pill-vs-solid debates.

Default CTA — hover me

Used for every primary action across the site: Send Message, View All, View Case Study, Next Project, etc. Text is always UPPERCASE. Arrow on the right slides 4px on hover. Each letter flips to accent yellow on hover with 20ms per-character stagger.

Disabled state

Faded white, no hover, arrow dimmed. Pointer-events disabled so hover animation doesn't fire.

Filter chip

Rectangular outlined chip for togglable filter groups. Sharp corners match the rest of the system. Active state: accent border and accent text, no fill.

06 — Forms

Input fields & form elements

Inputs should feel like part of the same family as buttons — same radius, same padding math, same focus ring.

Current — 12px radius, varied states

Inputs don't match buttons

Contact form uses 12px radius + 14px 18px padding. Filter chips use 100px. No consistency.

Enter a valid email address

Issues: Inputs at 12px radius, buttons at 999px or 100px. Error color uses hardcoded hex. Labels and inputs don't share a grid.
Improved — sharp, tokenized

Zero radius. Clean focus state.

Inputs match the rest of the system: sharp corners, subtle border. Focus turns the border accent yellow. Error state uses the --state-error token with a faint background tint.

Enter a valid email address

Rationale: Inputs feel like part of the editorial page, not a separate UI layer. Focus ring is always accent color — predictable. Error state uses --state-error + 4% background tint for soft context.

Every animation picks from a short list. One token per intent (hover, reveal, long-form, marquee).

Motion rules

Do

  • Use --duration-fast for hover (color, border, shadow)
  • Use --duration-base for movement (transform, slide)
  • Use --duration-slow for reveals (entry animations)
  • Respect prefers-reduced-motion — skip all non-essential animation
  • Trigger reveals via IntersectionObserver with threshold 0.15
  • Run each reveal once per scroll session (unobserve after)

Don't

  • Use arbitrary durations like 250ms, 320ms, 450ms — pick from the token list
  • Write inline cubic-bezier curves — use the 4 named easings
  • Add bounce/overshoot to navigational elements (feels gimmicky)
  • Trigger the same reveal twice if user scrolls up and back down
  • Use ease-in — entrances feel slow at the start (bad)
  • Stack more than 2 animations on one element at once
